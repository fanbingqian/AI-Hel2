> 版本：V1.0 | 2026-04-29
> 从 HEIMDALL 到知识环的统一架构方案

---

![知识环全貌 - 概念图]

## 一、设计哲学

### 1.1 一句话定位

**知识环是一个 AI 驱动的个人知识管理系统——用户不需要学习任何概念，不需要打标签、画关系图，所有积累在与 AI 的日常对话中自动完成。**

面向用户：非程序员的普通人
核心承诺：你的知识库每天都在自然增长，就像花园在长大

### 1.2 从 HEIMDALL 到知识环的故事

HEIMDALL 是为 AI Agent 设计的记忆系统——它帮 AI 记住你是谁、学过什么、和谁聊过。但它的存在是为了 AI，不是为用户。

知识环是 HEIMDALL 的"对用户端"——同一个数据、同一套引擎，但用户看到的不是数据库表，而是一个**每天都在变大的知识星球**。

```
HEIMDALL                         知识环
┌──────────────────────┐         ┌──────────────────────┐
│  AI 的记忆系统        │         │  用户的数字花园      │
│                      │         │                      │
│  · entity_store      │  ────►  │  · 环图可视化        │
│  · social_graph      │  同一   │  · 每日增长日报      │
│  · knowledge_entries │  数据   │  · 领域全景雷达      │
│  · persona_manager   │  库     │  · 社交森林          │
│  · ...               │         │  · 里程碑墙          │
└──────────────────────┘         └──────────────────────┘
```

### 1.3 核心原则

| 原则 | 含义 |
|---|---|
| **零学习成本** | 用户不需要理解实体、关系、标签等概念 |
| **自然积累** | 日常对话即输入，无需刻意"记录" |
| **景观化展示** | 用户看到的是"花园在长大"而不是数据表格 |
| **用户主权** | 所有数据归用户，可改可删可导 |
| **三层可见性** | AI看到的 ≠ 用户看到的 ≠ 用户想查的 |

## 二、用户端产品设计

### 2.1 命名与视觉隐喻

**命名：知识环（Knowledge Ring）**

视觉隐喻不是"图"而是"环"——围绕用户不断扩展、循环、生长的结构。用户站在中心，他的知识世界在一圈一圈向外扩张。

### 2.2 三层数据架构（对用户可见）

```
你说的话 → 统一数据入口
  │
  ├─ 🧑 用户画像（偏好/习惯/身份/社交/情绪）
  │   → tag: ai_only（AI内部用，不展示在环上）
  │   → 示例：「四四喜欢踏青」「小米用的手机」
  │
  ├─ 🧬 知识库（概念/人物/内容/作品/项目 + 关系 + 领域 + 掌握度 + 时间线）
  │   → tag: user_visible（展示在环上）
  │   → 示例：「Transformer」「王磊」「磁州窑方案」
  │
  └─ 📄 原文归档（完整原始文件）
      → tag: user_can_request（用户主动要才给）
      → 示例：一篇报告PDF、聊天记录长段落
```

**来源定位**：
- 🧑 画像 = 关于**你**的数据（AI帮你了解自己）
- 🧬 知识 = 关于**世界**的数据（AI帮你整理所学）
- 📄 原文 = 知识**来自哪里**（需要时回溯验证）

### 2.3 5类实体 + 8类关系

#### 实体类型

| # | 实体类型 | 例子 | 环上视觉 | 说明 |
|---|---|---|---|---|
| 🌿 | 概念/知识 | Transformer、供需缺口、B40政策 | 蓝色圆形 | 抽象的知识点、理论、概念 |
| 📄 | 内容/信息 | 一篇报告、一篇文章、一部电影 | 绿色方形 | 具体的信息载体 |
| 👤 | 人/角色 | 王磊、USDA、张总 | 橙色星形 | 自然人、机构、角色 |
| 📅 | 事/活动 | 4/29甲方会议、2025用户大会 | 红色三角 | 事件、活动、里程碑 |
| 🏺 | 物/作品 | 磁州窑展厅方案、Python脚本 | 紫色六边形 | 产出物、工具、作品 |

#### 关系类型

| # | 关系 | 方向 | 环上视觉 | 说明 |
|---|---|---|---|---|
| 1 | 属于 | 子→父 | 实线 | 子概念属于父领域 |
| 2 | 包含 | 父→子 | 实线 | 作品包含多个知识点 |
| 3 | 相关 | 双向 | 蓝色虚线 | 松散关联 |
| 4 | 对比/矛盾 | 双向 | 红色虚线 | 知识冲突 |
| 5 | 引起/导致 | 单向 | 红色箭头 | 因果关系 |
| 6 | 产出 | 单向 | 带箭头线 | 活动产出作品 |
| 7 | 启发 | 单向 | 带箭头线 | 作品启发新概念 |
| 8 | 认识/合作 | 双向 | 发光细线 | 社交关系 |

### 2.4 别名与消歧

| 问题 | 解决方案 |
|---|---|
| 同人多名：「王总」「王磊」「老王」「磊哥」 | 统一主 ID + 别名表 + 场景标记 |
| 同名多人：「小明」可能是用户自己也可能是别人 | 语义消歧 + 二次确认卡片 |
| 会话内省略：「他」「这哥们」「那个人」 | 当前会话活跃栈跟踪 |
| 跨会话识别 | 全局别名映射表 + 置信度分级 |

**二次确认在「理一理」环节**以卡片形式汇总，用户每天 0-5 次点击确认，不打断使用流程。

### 2.5 置信度系统（三色标记）

| 颜色 | 范围 | 行为 |
|---|---|---|
| 🔵 蓝色 | ≥ 85% | 自动连线，展示在环上 |
| 🟡 黄色 | 50% – 85% | 连线 + 标记「待确认」 |
| 🟠 橙色 | < 50% | 推给用户确认后再连 |

初始所有从 HEIMDALL 迁移的实体标记为 🟠 或 🟡，随着用户使用逐渐变为 🔵。

### 2.6 容错与纠错

| 机制 | 说明 |
|---|---|
| 三层自检 | 逻辑合理性 → 图谱结构异常 → 偏离用户历史模式 |
| 一键纠错 | 任何实体右上角 ✎ 可修改：分类/关联/合并/编辑/删除 |
| 拆分机制 | 合并错了可一键拆分，关联关系继承可选 |
| 自动学习 | 每个纠错 → AI 记住模式，下次不再犯 |
| 版本历史 | 类似 git reflog，可回溯修改记录 |
| 冲突处理优先级 | 用户手动操作 > 时间戳 > 置信度 |

### 2.7 用户场景（5大场景覆盖90%用户）

| 场景 | 触发 | 用户操作 | AI行为 |
|---|---|---|---|
| 🌱 随手记 | 聊完一个话题 | 无操作 | 自动提取实体+关系+领域 |
| 🔧 松土（理一理） | 每日/每周 | 确认0-5张卡片 | 去重、消歧、推荐新关联 |
| 👀 逛花园 | 想看看自己的知识 | 打开环图 | 展示已确认的知识景观 |
| 🔍 找东西 | 记得聊过但忘了 | 自然语言搜索 | 全文+图谱联合检索 |
| 📤 知识输出 | 需要产出 | 选领域/AI推荐 | 生成摘要、报告、卡片 |

## 三、技术架构

### 3.1 统一数据模型

#### 实体主记录

```
{
  "entity_id": "concept_0429_001",         // 全局唯一
  "type": "concept",                       // concept/person/content/event/artifact
  "canonical_name": "东南亚旱季延长",
  "aliases": [                             // 别名映射
    {"name": "气候异常", "context": "报告原文"},
    {"name": "厄尔尼诺", "context": "讨论"}
  ],
  "mastery": 0.7,                          // 0-1，用户掌握度（空则未知）
  "domain": ["气候", "农业"],               // 所属领域（可跨领域）
  "timeline": [                            // 时间线（原事件通道内容）
    {"date": "2026-04-29", "event": "首次记录", "status": "new"},
    {"date": "2026-04-30", "event": "关联到食用油报告", "status": "enriched"}
  ],
  "source_ref": "layer1://session_0429/paragraph_3",  // 指向原文
  "confidence": 0.92,                      // 置信度
  "tags": ["user_visible"],                // ai_only / user_visible / user_can_request
  "status": "active",                      // active / dormant
  "created_at": 1714372200,
  "updated_at": 1714458600
}
```

#### 关系记录

```
{
  "relation_id": "rel_0429_001",
  "source": "concept_0429_001",              // 东南亚旱季延长
  "target": "concept_0429_002",              // 棕榈油供给收缩
  "type": "cause",                           // 见8类关系
  "direction": "source->target",
  "confidence": 0.92,
  "source_ref": "[原文指针]",
  "tags": ["user_visible"],
  "created_at": 1714372200,
  "updated_at": 1714372200
}
```

#### 别名映射表

```
{
  "alias": "张总",
  "canonical_id": "person_wanglei_001",
  "context": "工作场合",
  "confidence": 0.85,
  "last_used": 1714372200
}
```

#### 用户画像条目

```
{
  "user_id": "ss",
  "trait": "户外活动偏好",
  "value": "喜欢踏青",
  "trigger": "天气好的时候",
  "confidence": 0.9,
  "tags": ["ai_only"],
  "created_at": 1714372200
}
```

### 3.2 AI执行逻辑：存入时序

```
你说了一句话
  │
  ├─ AI判断类型
  │   ├─ 闲聊/问候 → 不存
  │   ├─ 提问 → 走查询流程
  │   ├─ 指令 → 执行，不存（但产生的决策事件可存）
  │   ├─ 纠错/反馈 → 更新+同步
  │   ├─ 信息输入 → 进入存入流程
  │   └─ 隐含偏好表达 → 存入用户画像
  │
  ▼
存入流程（信息输入时）：
  Step 1: 语义拆解 → 段落→语义单元+语气标记
  Step 2: 实体提取 → 实体候选+类型推测
  Step 3: 消歧 → 查别名映射表→已有/新建
  Step 4: 关系构建 → 三元组+置信度+原文指针
  Step 5: 领域归位 → 归到已有领域/新建领域
  Step 6: 写入统一数据库
        ├─ 实体记录（含时间线更新）
        ├─ 关系记录
        ├─ 别名映射（如有新别名）
        └─ 原文归档（如输入包含原文）
```

### 3.3 AI执行逻辑：查询时序

```
你问了一个问题
  │
  ├─ 问题类型判断
  │   ├─ 精确事实型（"B40是什么？"）
  │   │   → 查实体记录 + 关系记录（tag: user_visible）
  │   │
  │   ├─ 历史回顾型（"上次那篇报告说什么？"）
  │   │   → 先查对话摘要 + 找到实体ID
  │   │   → 再查实体记录的时间线 + 关系
  │   │
  │   ├─ 模糊联想型（"好像聊过树龄的事？"）
  │   │   → 查全文索引 + 别名映射表 + 关系路径
  │   │
  │   ├─ 要原文型（"把报告原文发我"）
  │   │   → 查实体记录 → 取 source_ref
  │   │   → 从原文存储提取文件，AI不总结直接给
  │   │
  │   ├─ 推荐型（"今天去哪踏青？"）
  │   │   → 查用户画像（tag: ai_only），结合偏好回答
  │   │
  │   └─ 修改数据型（"改一下这个"）
  │       → 查实体ID → 修改字段 → 更新数据
  │       → 不需要同步（就这一份数据）
```

### 3.4 存储方案

| 数据 | 推荐储存 | 备选方案 | 索引方式 |
|---|---|---|---|
| 实体主记录 | 图数据库(Neo4j/ArangoDB) | PostgreSQL | 实体ID、类型、领域、别名 |
| 关系 | 图数据库 | PostgreSQL + 边表 | source、target、type |
| 别名映射表 | 图数据库属性 / Redis | PostgreSQL | 别名全文索引 |
| 时间线 | 实体属性嵌套数组 | 独立时间线表 | 时间范围查询 |
| 用户画像 | 文档数据库(MongoDB) | PostgreSQL jsonb | 用户ID + 分类 |
| 原文归档 | 对象存储(S3/本地FS) | SQLite blob | 文件ID、会话ID、日期 |
| 全文检索 | Elasticsearch | SQLite FTS5 | 全文索引 |

**推荐轻量方案**：全量使用 PostgreSQL + pgvector，兼顾关系查询+全文检索+向量搜索，减少运维复杂度。

### 3.5 技术栈选型建议

| 层级 | 选择 | 理由 |
|---|---|---|
| 数据库 | PostgreSQL + pgvector | 关系、全文检索、向量搜索三合一 |
| 搜索引擎 | 内置 PostgreSQL FTS / elasticsearch | 小规模用PG FTS，大规模单独部署 |
| 原文存储 | 本地文件系统 + 索引表 | 个人使用，无需S3 |
| 前端渲染 | D3.js / vis.js | 环图可视化，开源社区成熟 |
| 图算法 | NetworkX / 数据库内算法 | PageRank、社区发现 |
| 后端 | Python (FastAPI) | 与AI栈一致 |
| 前端框架 | React / Vue | 用户界面 |

## 四、与当前 HEIMDALL 的映射关系

### 4.1 差异分析

| 能力维度 | 当前 HEIMDALL | 知识环目标 | 差距 |
|---|---|---|---|
| 实体 | ✅ 9 种类型 + 属性 + FTS | ✅ 5 种类型 + 属性 + FTS | 精简类型 |
| 关系 | ⚠️ 只有 social_graph + memory_edges | ✅ 8 类语义关系 | ❗需要新建`knowledge_ring_edges` |
| 社交 | ✅ 强度/效价/波动/健康分 | ✅ 可作为`认识/合作`关系的属性 | 映射即可 |
| 别名 | ❌ 无 | ✅ 别名映射表 + 消歧 | ❗新建 |
| 领域分类 | ❌ 仅有 domain 字符串 | ✅ 自动归位 + 新领域发现 | ❗增强多领域支持 |
| 置信度 | ✅ 实体/知识 有 confidence | ✅ 三色制 | 保留，增加UI标记 |
| 用户画像 | ✅ Persona Manager | ✅ tag: ai_only | 保留不变 |
| 技能掌握度 | ✅ 独立表 + knowledge 关联 | ❌ 改为实体 attribute `mastery` | 迁移 |
| 时间线 | ❌ 仅有 created_at/updated_at | ✅ 实体 timeline 数组 | ❗新建 |
| 原文归档 | ❌ 无 | ✅ layer1 存储 | ❗新建 |
| 可视化 | ❌ 无 | ✅ 环图 + 日报 + 周报 | ❗新建 |
| 纠错 | ⚠️ oral correction only | ✅ 一键纠错 + 拆分 + 自动学习 | ❗新建 |
| 版本历史 | ⚠️ audit table for edges | ✅ 所有操作可追溯 | 扩展审计范围 |
| 事件 | ⚠️ event entity type exists | ✅ 合并到实体的 timeline 属性 | 迁移 |

### 4.2 迁移方案：HEIMDALL 到知识环

```
当前状态：
  heimdall_entities  ──►  知识环实体（字段扩充，类型映射）
  heimdall_social_graph ──► 知识环社交关系（映射到认识/合作关系）
  heimdall_knowledge_entries ──► 知识环实体（转为实体 + 掌握度 attribute）
  heimdall_skill_mastery ──► 知识环实体 attribute（mastery 字段）
  heimdall_memory_edges ──► 保留（会话关联，不展示）
  heimdall_persona ──► 用户画像通道（tag: ai_only）
  
新建：
  knowledge_ring_aliases ──► 别名映射表
  knowledge_ring_edges ──► 8类语义关系表
  knowledge_ring_originals ──► 原文归档
  knowledge_ring_timeline ──► 合并到实体的 timeline 字段
```

**迁移规则**：
1. 现有实体类型映射：`skill→concept`, `event` timeline 属性迁移, `media` 保留但加 tag
2. 所有迁移实体初始置信度设为 🟡（50-85%）
3. 社会关系自动映射为`认识/合作`关系
4. 知识条目转为实体 + mastery 字段
5. 不破坏现有 HEIMDALL API，知识环新增的表和字段为扩展

### 4.3 向后兼容

所有现有 HEIMDALL 工具（heimdall_persona、heimdall_knowledge、heimdall_memory）**继续正常工作**。知识环作为上层扩展，新增工具：

```
heimdall_ring ──► 知识环专用工具
  ├─ search_by_domain: 按领域搜索
  ├─ get_daily_growth: 获取每日增长数据
  ├─ get_entity_relations: 获取实体关联全景
  ├─ correct_entity: 一键纠错
  ├─ split_entity: 拆分实体
  ├─ merge_entities: 合并实体
  └─ get_timeline: 获取实体时间线
```

## 五、实现路线图

### Phase 1：数据层扩展（1-2周）

```
目标：在现有 HEIMDALL 上建立知识环的数据基础

P0：
  □ 新建 knowledge_ring_edges 表（8类关系 + 方向 + 置信度 + tag）
  □ 新建 knowledge_ring_aliases 表（别名映射 + 上下文 + 置信度）
  □ 为已存在实体增加 timeline 字段（利用现有时间戳构建初始时间线）
  □ 新建 heimdall_ring 工具（以上表的基本CRUD）

P1：
  □ 将现有 social_graph 数据映射到 knowledge_ring_edges
  □ 将 knowledge_entries + skill_mastery 合并为实体 mastery 属性
  □ 新建原文归档存储（目录结构 + 索引表）
```

### Phase 2：AI逻辑增强（1-2周）

```
目标：AI在对话中自动完成实体提取、消歧、关系构建

P0：
  □ 存入时序实现（Step 1-6 管道）
  □ 别名消歧引擎（会话栈 + 全局映射 + 置信度）
  □ 领域自动归位（Few-shot 分类器）

P1：
  □ 三层自检机制（合理性→结构异常→偏离模式）
  □ 二次确认卡片生成
  □ 自动学习（纠错→更新规则）
  □ 掌握度自动评估
```

### Phase 3：展示层（2-3周）

```
目标：用户能看见自己的知识花园

P0：
  □ 环图渲染引擎（D3.js / vis.js）
  □ 每日/每周增长播报
  □ 领域全景雷达图
  □ 实体卡片展示（含时间线、关联、掌握度）

P1：
  □ 交互式环图（缩放、拖拽、筛选领域）
  □ 一键纠错 UI
  □ 社交森林视图
  □ 里程碑墙
  □ 月度/年度总结报告
```

### Phase 4：优化与扩展（持续）

```
目标：知识环成为用户日常使用的核心工具

  □ "随手记"场景 90% 以上自动提取准确率
  □ 全文检索 + 图谱联合搜索
  □ 知识输出（自动生成摘要、报告、卡片）
  □ 多端同步（桌面 + 移动）
  □ 数据导出（JSON / Markdown / GraphML）
  □ 与第三方工具集成（Notion / Obsidian / Logseq）
```

## 六、关键设计决策记录

| # | 决策 | 结论 | 原因 |
|---|---|---|---|
| 1 | 产品名称 | **知识环（Knowledge Ring）** | 简洁、有记忆点、"环绕/循环/生长" |
| 2 | 与HEIMDALL关系 | **融合为一个系统** | 消除数据冲突，简化AI查询 |
| 3 | 技能通道 | **取消，改为实体属性** | 独立通道与知识通道重叠 |
| 4 | 事件通道 | **与知识通道合并** | 事件本质是实体的时间维度 |
| 5 | 通道总数 | **3个入口：画像+知识+原文** | 覆盖95%以上场景 |
| 6 | 用户画像可见性 | **tag: ai_only** | 隐私保护，不干扰知识视图 |
| 7 | 原文可见性 | **tag: user_can_request** | 避免信息过载，需要时可取回 |
| 8 | 实体类型数 | **5类** | 足够区分主要知识形态，不过度 |
| 9 | 关系类型数 | **8类** | 覆盖常见语义，用户可理解 |
| 10 | 别名消歧 | **全局映射表 + 会话栈** | 跨会话+同会话内指代消解 |
| 11 | 置信度 | **三色标记** | 降低用户认知负担 |
| 12 | 纠错优先级 | **用户操作 > 时间戳 > 置信度** | 用户主权 |
| 13 | 存储 | **PG + pgvector 推荐** | 关系+全文+向量三合一 |
| 14 | 向后兼容 | **HEIMDALL API 不变** | 渐进式迁移，不断已有工具 |

## 七、术语表

| 术语 | 解释 |
|---|---|
| 知识环（Knowledge Ring） | 产品名称，用户的数字知识花园 |
| HEIMDALL | AI 记忆系统，知识环的后端引擎 |
| 实体（Entity） | 知识的最小单位：概念、人、内容、事件、物 |
| 关系（Edge） | 两个实体之间的语义连接 |
| 领域（Domain） | 知识的上层分类，如"AI/ML"、"产品设计" |
| 别名（Alias） | 同一实体的不同称呼 |
| 置信度（Confidence） | AI 对实体/关系准确性的把握程度 |
| 环图（Ring View） | 知识环的默认可视化样式 |
| 松土（Tending） | 用户确认/纠错知识的环节 |
| 档案层（Layer1） | 原文归档，知识的原始来源 |

| 纠错「这个是概念不是人」 | ✅ 修改类型，AI 记住 | ✅ |
| 情绪表达「好烦啊今天」 | ✅ 情绪状态存入画像 | ✅ |
| 元讨论「我们刚才设计的不错」 | ✅ 存入事件，关联设计决策实体 | ✅ |
