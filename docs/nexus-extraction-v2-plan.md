# Nexus 知识提取与图谱渲染 v2 方案

> 基于文档优先的知识图谱架构，重新设计提取逻辑和图谱交互

## 一、核心设计原则

```
文档是一等公民。所有知识从文档中提取，文档之间可以跳跃。
图谱默认折叠显示文档关系网，展开后看到每个文档的内部知识结构。
```

## 二、实体提取规则

### 第一优先级：文档实体（必须）

| 规则 | 说明 |
|------|------|
| 每个文档 = 1 个独立实体节点 | type: `document`, name: 文件名(去扩展名) |
| 无论内容是否为空 | 文件存在即创建 |

### 第二优先级：命名实体（必须）

| 类型 | 示例 | 识别方式 |
|------|------|---------|
| 地名 | 巴厘岛、金巴兰湾、京打马尼火山 | NER / LLM |
| 企业/组织名 | 长实集团、阿雅娜度假酒店 | NER / LLM |
| 人名 | 李嘉诚、刘銮雄 | NER / LLM |
| 自然名 | Rock Bar 岩石酒吧、科莫多国家公园 | NER / LLM |

命名实体**不受数量限制**，识别到就提取。

### 第三优先级：信息实体（筛选提取）

通过四步检查筛选，每步取前 3 个：

#### 检查 1：信息密度检查（Top 3）
```
判断标准:
  - 是否具体、可独立存在？
  - 去掉它，文段还完整吗？
  - 它携带了多少信息量？

示例:
  ✅ "知识架构一体化方案" — 具体、独立、信息量大
  ✅ "per-agent namespace" — 独特概念、不可删除
  ❌ "容器化" — 太泛、可被更具体实体替代
```

#### 检查 2：分散度检查（Top 3）
```
判断标准:
  - 它在文档中均匀出现还是只在一处提及？
  - 贯穿全文 → 核心主题
  - 集中一处 → 局部细节

示例:
  ✅ 全文多处讨论"Docker部署" → 核心主题
  ❌ 只在文末提了一次"Jenkins" → 局部细节
```

#### 检查 3：独特性检查（Top 3）
```
判断标准:
  - 它是行业通用术语还是用户特有的概念？
  - 独特 → 高价值知识
  - 通用 → 降权但仍可提取

示例:
  ✅ "巴厘岛8天7晚定制方案" — 用户特有，独特性高
  ❌ "Docker" — 行业通用，独特性低
```

### 第四优先级：关联性检查

#### A. 文档与文档的关系（LLM 写入）

```
LLM 在提取完成后，分析当前文档与其他已知文档的关系:
  - 延续/扩展: "A 是 B 的详细方案"
  - 引用/参考: "A 引用了 B 的架构设计"
  - 主题相关: "A 和 B 都讨论知识引擎"
  - 时间顺序: "A 是 B 的前置研究"

关系写入: cache_relations (from=docA, to=docB, type=extends/refers/related_to/precedes)
```

#### B. 实体与实体的关系

```
已提取实体之间:
  - A uses B
  - A depends_on B
  - A contains B
  - A located_in B
  - A creates B

实体与原文档之间:
  - doc contains entity (默认关系)
  - doc discusses entity
```

## 三、图谱渲染设计

### 默认视图：文档关系网（社区折叠）

```
┌─────────────────────────────────────────┐
│  图谱默认状态（启动时）                    │
│                                          │
│   [巴厘岛旅游.md] ──extends──→ [科莫多.md] │
│        │                    │             │
│        │ related_to         │ refers      │
│        ↓                    ↓             │
│   [知识架构方案.md]    [HEIMDALL.md]       │
│                                          │
│  只显示:                                  │
│    - 文档节点（清晰可辨的方形或圆角）        │
│    - 文档之间的连线（有则连，无则孤岛）       │
│    - 孤岛文档独立显示，无连线               │
└─────────────────────────────────────────┘
```

### 展开视图：文档内部知识（点击/双击展开）

```
┌─────────────────────────────────────────┐
│  点击 [巴厘岛旅游.md] 展开后:              │
│                                          │
│         [巴厘岛旅游.md] ← 中心文档         │
│         /    |     \                     │
│        /     |      \                    │
│   [巴厘岛]  [金巴兰湾]  [阿雅娜]           │
│   (地名)   (地名)     (组织)              │
│     |        |         |                 │
│   [Rock Bar] [Akasa]  [京打马尼]          │
│   (自然名)   (地名)    (自然名)            │
│     |                                   │
│   [8天7晚定制] ← 独特性Top3               │
│   [悬崖之巅体验] ← 信息密度Top3            │
│                                          │
│  显示:                                    │
│    - 文档节点（带展开标记）                 │
│    - 命名实体（地名/组织/人名/自然名）       │
│    - 信息实体（信息密度Top3 + 分散度Top3    │
│      + 独特性Top3 去重后合并）              │
│    - 所有关系线                           │
└─────────────────────────────────────────┘
```

### 交互行为

| 操作 | 效果 |
|------|------|
| 图谱默认加载 | 社区折叠视图（只有文档节点和文档间连线） |
| 单击文档节点 | 选中，右侧显示文档信息 |
| 双击文档节点 | 展开该文档的内部知识结构 |
| 双击展开的文档节点 | 收起，回到文档网络视图 |
| 拖拽节点 | 重新布局 |
| 滚轮 | 缩放 |
| 搜索 | 可以搜索文档名或实体名 |

### 配置面板：社区折叠开关

在 GraphSettingsPanel 筛选区新增「社区折叠」开关：

| 开关状态 | 效果 |
|---------|------|
| **开启**（默认） | 全部文档折叠 → 只显示文档节点 + 文档间连线；孤岛文档独立显示 |
| **关闭** | 全部文档展开 → 显示所有文档节点 + 命名实体 + 信息实体 + 全部关系线 |

开关一键控制全部文档的展开/折叠。单个文档的双击操作仍然独立生效：
- 全局折叠模式下双击某文档 → 该文档临时展开（其他保持折叠）
- 全局展开模式下双击某文档 → 该文档临时折叠（其他保持展开）

## 四、提取流程

```
新文档写入 wiki/
  ↓
① 创建文档实体 (type: document, id: doc:{path})
  ↓
② NER 提取命名实体 (地名/组织/人名/自然名)
  → 无数量限制，识别到就创建
  → 创建 contains 关系到文档
  ↓
③ LLM 提取信息实体
  → 四步检查（信息密度/分散度/独特性/关联性）
  → 每步取 Top 3，去重后合并
  → 创建对应关系到文档
  ↓
④ 关联性检查 - 文档关系
  → LLM 分析本文档与已有文档的关系
  → 写入文档间关系 (extends/refers/related_to/precedes)
  ↓
⑤ 关联性检查 - 实体关系
  → 已提取实体之间的关联
  → 跨文档同名实体自动关联（同 ID）
```

## 五、与现有系统的改动

| 模块 | 改动 | 说明 |
|------|------|------|
| `extract_service.py` | 重写 prompt | 四步检查框架 + 命名实体优先 |
| `knowledge_service.rs` | 新增文档间关系提取 | LLM 写文档关系 |
| `graphAdapter.ts` | 新增文档模式 | 默认只显示文档节点 |
| `ForceGraph2DWrapper.tsx` | 新增展开/折叠 | 双击展开文档内部 |
| `knowledge.ts` | 新增字段 | `is_document_node`, `document_id` |
| 数据库 | 无 schema 变更 | `__file__` 类型的 entity 即文档节点 |

## 六、命名实体类型约束

为避免类型碎片化，LLM 提取命名实体时使用固定类型名：

| 类型 | type 值 | 示例 |
|------|---------|------|
| 地名 | `location` | 巴厘岛、金巴兰湾 |
| 企业/组织 | `organization` | 长实集团、阿雅娜度假酒店 |
| 人名 | `person` | 李嘉诚 |
| 自然景观 | `natural_feature` | Rock Bar、京打马尼火山 |

其他 LLM 自由提取的实体类型不做限制，但命名实体必须使用以上四种。

## 七、数据结构

### 文档间关系

```json
{
  "from_id": "doc:巴厘岛旅游.md",
  "to_id": "doc:科莫多旅游.md",
  "relation_type": "extends",
  "label": "巴厘岛行程延伸至科莫多",
  "confidence": 0.8
}
```

### 关系类型枚举

| 类型 | 含义 |
|------|------|
| `extends` | A 是 B 的延伸/详细方案 |
| `refers` | A 引用了 B |
| `related_to` | A 和 B 主题相关 |
| `precedes` | A 是 B 的前置/依赖 |
