## 二、行业最佳实践参考（2024-2025）

### Semantic Entity Resolution (SER)

LLM 驱动的实体解析已成为 2024-2025 的主流方案。核心流程：

```
Blocking → Matching → Merging → Canonicalization
(向量聚类)  (LLM判断)   (合并)    (规范化)
```

关键原则：
- **Auto-merge threshold**: ≥0.98 自动合并；0.90-0.97 标记待审核；<0.90 保留为新实体
- **Cross-type matching**: 不同 entity_type 但语义相同也应匹配
- **SAME_AS 边**: 匹配后创建等价关系，形成连通分量后再批量合并
- **Provenance**: 每个合并事实保留来源引用

### Reflection-based Agentic Workflow

```
提取 → 批评(验证) → 修正 → 再验证
       ↑ 检查类型一致性、schema合规、矛盾标记
```

实验证明 schema 合规率从 95.9% 提升到 98.1%。

## 三、v2 维护操作重构方案

### 合并为 6 个操作组

```
操作组 A: 知识库健康检查     ← 原: 质量评分 + 孤岛清理 + 修复迁移
操作组 B: 实体去重与合并     ← 原: 去重检查（重构为跨类型语义匹配）
操作组 C: 文档归类           ← 原: 增量 + 全量（保留，提升优先级）
操作组 D: 图谱结构分析       ← 原: PageRank + 社区检测
操作组 E: 关系推导与验证     ← 原: 传递推理 + 验证合成边
操作组 F: 冲突与矛盾检测     ← 原: 冲突扫描（扩展到全量关系对）
```

---

### 操作组 A: 知识库健康检查

**合并**: 质量评分 + 孤岛清理 + 修复迁移

```
流程:
  1. 规则检查（不调 LLM）
     - 文档实体检查: source_file 是否存在？
     - 命名实体检查: 是否有关联文档？
     - 信息实体检查: 描述是否为空？置信度是否 < 0.4？
     - 孤立检查: 文档节点不算孤岛，非文档节点无关系才算
     - 类型引号清理（原迁移修复）

  2. 自动修复
     - 过期（源文件已删）→ 标记 hidden=1 或标记 stale
     - D级低质量 → hidden=1
     - 类型引号 → 去掉

  3. 输出
     - 健康度报告: 文档数/命名实体数/信息实体数/关系数
     - 各等级数量（A/B/C/D）
     - 待处理项列表
```

### 操作组 B: 实体去重与合并

**重构**: 从"同名同类型"升级为"跨类型语义匹配"

```
流程:
  1. Blocking（向量聚类）
     - 用 name 的字符相似度 + entity_type 进行预分块
     - 减少 O(n²) 比较

  2. Matching（LLM 批量判断）
     - 每批提交 5-10 对候选给 LLM
     - LLM 输出: { match: true/false, confidence: 0.0-1.0, reason }
     
  3. Merge
     - confidence ≥ 0.95 → 自动合并
     - 0.85-0.95 → 创建 SAME_AS 边，标记待审核
     - < 0.85 → 保留为独立实体

  4. Canonicalization
     - 合并后选择最佳名称（最长/最具体/最完整）
     - 旧名保留到 aliases[]
     - 关系全部转移到新实体
```

### 操作组 C: 文档归类

**提升**: v2 中文档是一等公民，归类操作更重要

```
流程:
  增量: 扫描 wiki 根目录未分类文件 → LLM 建议文件夹 → 移动
  全量: 遍历整个 wiki 树 → LLM 分类 → 移动

增强: 归类后自动触发文档关系提取（操作组扩展）
```

### 操作组 D: 图谱结构分析

**保留**: PageRank + 社区检测

```
PageRank (有向图迭代):
  - d=0.85, max 100 轮
  - 输出 Top N 核心实体
  - v2 增强: 分别计算文档级和实体级 PageRank

社区检测 (Louvain):
  - 自动发现紧密关联的实体群组
  - v2 增强: 社区结果用于图谱默认折叠视图
```

### 操作组 E: 关系推导与验证

**合并**: 传递推理 + 验证合成边

```
流程:
  1. 传递推理 (规则, 不调 LLM)
     - 扫描 A→B→C 传递链
     - 推断 A→C (confidence = min(w1,w2) × 0.9, capped 0.5)
     - 推断边标记 inferred=1

  2. 验证合成边 (调 LLM)
     - 对所有 inferred=1 的边分批验证
     - LLM 判断合理性 → 确认保留 / 拒绝删除
     - 输出验证报告

v2 增强: 扩展传递关系类型（从4种增加前缀匹配到10种）
```

### 操作组 F: 冲突与矛盾检测

**扩展**: 从 3 对硬编码到扫描全部互斥模式

```
当前: 只查 3 对 (supports↔opposes, agrees↔disagrees, proves↔disproves)
扩展: 增加语义矛盾检测
  - 双向不对称: A contains B 但 B contains A
  - 循环依赖: A depends_on B depends_on C depends_on A
  - 时间矛盾: A precedes B 但 B 的时间戳早于 A
  - LLM 辅助判断新型矛盾对
```

```

┌─ 知识库健康检查 ────────────────────────────────────┐

├─ 关系推导与验证 ────────────────────────────────────┐
```
